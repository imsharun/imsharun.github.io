## CI / Deployment

This repo includes a GitHub Actions workflow `.github/workflows/deploy.yml` that builds the Vite frontend and deploys `dist/` to S3, then invalidates a CloudFront distribution.

Required repository Secrets (set under Settings → Secrets):

- `VITE_RAZORPAY_KEY` — your public Razorpay key (starts with `rzp_`).
- `VITE_API_BASE_URL` — backend API base URL (e.g. `https://api.example.com`).
- `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` — AWS credentials for deployment.
- `AWS_REGION` — AWS region (e.g. `us-east-1`).
- `S3_BUCKET` — target S3 bucket name for static assets.
- `CLOUDFRONT_DISTRIBUTION_ID` — CloudFront distribution id to invalidate after deploy.

Security notes:

- Only `VITE_`-prefixed env vars are inlined into the built frontend. Keep private secrets (Razorpay secret) on the backend / Secrets Manager.
- Do not commit `.env` files containing secrets; use GitHub Secrets for CI builds.

## AWS Console Pre-Setup

Before running the CI deploy you should create the AWS resources in the console (or via IaC). Minimal steps:

1. Create an S3 bucket for the site
  - Choose a globally-unique name (e.g. `my-oregano-site`).
  - Block public access; we will front the bucket with CloudFront (do NOT make it public).
  - Enable versioning (optional).

2. Request an ACM certificate for your domain
  - Region: **us-east-1** (required for CloudFront).
  - Validate via DNS (recommended) or email.

3. Create a CloudFront distribution
  - Origin: your S3 bucket.
  - Origin access: create an Origin Access Control (OAC) when prompted and allow CloudFront to update bucket permissions.
  - Default root object: `index.html`.
  - Attach the ACM certificate and set the alternate domain name (CNAME) if using a custom domain.
  - Configure caching: long TTLs for hashed assets, shorter TTL for `index.html`.

  If you don't have a custom domain yet you can use CloudFront's autogenerated domain instead:

  - CloudFront provides a domain like `dXXXXXXXXXXXX.cloudfront.net` and serves it over HTTPS with Amazon's certificate automatically.
  - To find the domain: Console → CloudFront → Distributions → open your distribution → read the **Domain name** field. Or with the CLI:

  ```bash
  aws cloudfront get-distribution --id DISTRIBUTION_ID --query "Distribution.DomainName" --output text
  ```

  - This is fine for testing or temporary public URLs. If you later buy a domain, request an ACM cert in `us-east-1`, attach it to the distribution, and point your DNS to the CloudFront domain.

4. (Optional) Configure Route53
  - Create an `A` / `AA` alias record that points your domain to the CloudFront distribution.

5. Create an IAM user for CI deployment
  - Give programmatic access (access key + secret) and attach a minimal policy allowing:
    - `s3:ListBucket`, `s3:GetObject`, `s3:PutObject`, `s3:DeleteObject` on the S3 bucket
    - `cloudfront:CreateInvalidation` on `*` (or on the distribution resource)
  - Save the `AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY` for GitHub Secrets.

6. (Optional) SES setup for sending email from Lambda
  - Verify sending domain or email in SES (region selection depends on your Lambda setup).
  - Request production access if you need to send to arbitrary addresses.

7. Configure CORS for your backend API
  - If your backend is API Gateway, enable CORS for your frontend origin or allow `*` for testing.

8. Add GitHub Secrets used by the workflow
  - `VITE_RAZORPAY_KEY`, `VITE_API_BASE_URL`, `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_REGION`, `S3_BUCKET`, `CLOUDFRONT_DISTRIBUTION_ID`.

After these console steps you can push to `main` and the workflow will build and publish the site.

### CLI commands for the above steps

1) Create S3 bucket, block public access, enable versioning:

```bash
# replace BUCKET_NAME and REGION
aws s3api create-bucket --bucket BUCKET_NAME --create-bucket-configuration LocationConstraint=REGION
aws s3api put-public-access-block --bucket BUCKET_NAME --public-access-block-configuration \
  BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true
aws s3api put-bucket-versioning --bucket BUCKET_NAME --versioning-configuration Status=Enabled
```

2) Request ACM certificate (us-east-1 for CloudFront):

```bash
# replace DOMAIN_NAME and validation-by DNS choice
aws acm request-certificate --domain-name DOMAIN_NAME --validation-method DNS --region us-east-1
```

3) (Optional) Sync built `dist/` to S3 (used by the workflow as well):

```bash
aws s3 sync dist/ s3://BUCKET_NAME --delete
```

4) Create CloudFront invalidation (after deploy):

```bash
aws cloudfront create-invalidation --distribution-id DISTRIBUTION_ID --paths "/*"
```

5) Create an IAM user for CI and generate access keys (CLI):

```bash
# create user
aws iam create-user --user-name deployer
# attach policy allowing S3 and CloudFront invalidation (example policy ARN or custom policy)
# create access keys
aws iam create-access-key --user-name deployer
```

6) SES (backend-handled)

- Email sending and SES verification are handled by the backend service/Lambda. No frontend CLI steps are required here. If you manage SES yourself, verify your sending domain or email in SES and request production access from the SES console.

Notes:
- Some operations (CloudFront distribution creation, OAC setup) are easier in the AWS Console; CloudFront create-distribution via CLI requires a JSON config and is more involved.
- When using the CLI, replace placeholders (`BUCKET_NAME`, `REGION`, `DOMAIN_NAME`, `DISTRIBUTION_ID`) with actual values.

## Shopping Cart App

- Overview: A simple shopping site with products, cart state, and routing.
- Features: Product grid, add/remove items, quantity updates, subtotal, persisted cart.

### Run Locally

1. Install dependencies:

```bash
npm install
```

2. Start dev server:

```bash
npm run dev
```

3. Build for production:

```bash
npm run build
npm run preview
```
